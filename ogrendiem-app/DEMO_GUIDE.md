# öğrenDiem — Demo Guide

A complete walkthrough for (re)generating the bundled data, running the
Expo app on a phone or the web, and scripting a compelling demo flow.

---

## 0. What this demo is

- **Scope:** chapters 1, 3, 8, 9 of Lippman & Rasmussen's *Precalculus*.
- **Data:** 55 topics, 140 prerequisite edges, 5 clusters,
  30 hand-curated bundled questions.
- **Model:** Bayesian Knowledge Tracing per topic + noisy-AND prereq
  propagation (JS port of the Python pgmpy reference).
- **UI:** four tabs — **Garden** (per-chapter tree), **Cave** (per-cluster
  cave), **Learn** (question loop), **Progress** (mastery + collection).
- **Offline-first:** all content bundled in `assets/data/*.json`; the entire
  adaptive engine runs on-device. A future `RemoteEngine` is a single-file
  swap (`src/engine/index.ts`).

## 2. One-time setup

From the repo root (`toplumsal-katki/`):

### 2a. Python side (only if you need to regenerate bundled data)

```bash
cd ogrendiem
python -m venv .venv
# Windows:
.\.venv\Scripts\activate
# macOS/Linux:
source .venv/bin/activate

pip install -r requirements.txt   # networkx, pgmpy, numpy, etc.
```

### 2b. App side

```bash
cd ogrendiem-app
npm install
```

This pulls Expo SDK 51, React Native 0.74.5, `react-native-svg`,
`react-native-webview`, `zustand`, `zod`, AsyncStorage, and navigation.

## 3. Regenerating bundled data (optional)

The repo already ships with `assets/data/{topics,graph,clusters,questions}.json`
populated for g1-3-8-9. You only need to regenerate if you:

- change the scope (e.g. add another chapter),
- tweak the emoji pools or cluster colors,
- re-run `parallelism.py` with different parameters.

```bash
cd ogrendiem
python -m graph.export_mobile
```

This reads the processed LibreTexts DAG, computes per-topic `garden_slot`
and `cave_slot`, runs the parallelism clustering, assigns deterministic
emojis by `sha256(topic_id) mod pool`, and writes four files into
`../ogrendiem-app/assets/data/`.

Expected log tail:

```
wrote topics.json   (55 topics)
wrote graph.json    (140 edges)
wrote clusters.json (5 clusters)
```

**Note:** `questions.json` is *not* regenerated by this script — it is the
hand-curated bundle of 30 LLM-authored worked examples. Editing it is a
manual step (see §7).

## 4. Running the app

You have three realistic targets. Pick one.

### 4a. Physical phone via Expo Go (recommended for demo)

```bash
cd ogrendiem-app
npm start
```

A QR code appears in the terminal.

- **Android:** open Expo Go → "Scan QR code" → aim at the terminal.
- **iOS:** open the Camera app → point at the QR → tap the banner.

The phone and the dev machine must be on the same Wi-Fi. If they're not
(lab Wi-Fi isolation, corporate VPN), press `s` in the Expo CLI to switch
to **tunnel** mode — it routes through ngrok at the cost of ~2 s extra
latency.

First launch bundles ~3–5 MB of JS; subsequent launches are instant.

### 4b. Android emulator

```bash
npm run android
```

Requires Android Studio with an AVD already running. Metro will install
the JS bundle into Expo Go inside the emulator.

### 4c. Web preview

```bash
npm run web
```

Opens `http://localhost:8081` (or `:19006`). **Caveats for web:**

- The KaTeX `WebView` inside `MathView.tsx` becomes an `<iframe>` — it
  works but layout measurement is a frame slower.
- Touch hit-slop on `EmojiNode` is tuned for fingers; on desktop you
  click a 44-px target just fine.
- `react-native-svg` renders via browser SVG, identical visuals.

Web is fine for presentations where a laptop + projector is easier than
miracle Wi-Fi.

### Troubleshooting startup

| Symptom | Fix |
|---|---|
| `Unable to resolve "@/…"` | `tsconfig.json` paths aren't applied — restart Metro with `npm start --clear`. |
| KaTeX shows raw `$x=3$` | WebView failed to load CDN. Reconnect to the internet or inline KaTeX as a local asset. |
| Empty Garden screen | `assets/data/topics.json` is stale. Re-run §3 or restore from git. |
| "Network request failed" on Expo Go | Phone and laptop on different networks. Press `s` for tunnel mode. |
| Metro stuck on "Building JS bundle" | `rm -rf node_modules .expo && npm install && npm start --clear`. |

## 5. The demo flow (5–7 minutes)

A tight script that shows every moving part without dragging.

### Beat 1 · "Here's the map" (30 s)
Open **Garden** → scroll chapter chips → explain each tree is one chapter,
each emoji is a topic, branches are sections. Flip to **Cave** → explain
each cave is a prerequisite cluster (found automatically by
`parallelism.py`), tunnels are in-cluster prereqs.

### Beat 2 · "The engine is adaptive" (60 s)
Tap **Learn** with no topic selected → the engine picks the frontier-best
topic via `getNextTopic(scope='group/g1-3-8-9')`. Point at the mastery
bar (0 %) and say: "all topics start at the BKT prior; the engine only
unlocks a topic once all its DAG parents cross 80 %."

### Beat 3 · "Answering moves the graph" (2 min)
On the Learn screen:
1. Hit "Show next step" a few times to reveal the scaffold.
2. Hit "Reveal answer".
3. Tap **I got it ✓**.

Two things happen, both visible:
- The mastery bar jumps (BKT posterior update).
- Go back to **Garden** — that topic's emoji now has a green **+1 stash**
  badge. That's the collectable: one correct answer = one emoji waiting.

Tap the badged emoji on the tree → stash goes to 0, a grey picked counter
appears at **1**. The mastery value is *unchanged* — collection and
mastery are decoupled on purpose.

### Beat 4 · "Prereqs propagate" (90 s)
Answer two or three questions on an early topic (e.g. `ch1_s1_t1` —
function notation). Open **Progress** and scroll to a downstream topic
(e.g. `ch1_s4` — composition of functions). Its bar has *also moved*,
even though the student never answered it, because noisy-AND
propagation lifts descendants when a prereq improves.

### Beat 5 · "Frontier unlock" (45 s)
Keep pushing mastery on chapter-1 basics until one downstream topic's
parents all cross 0.80. Tap **Next recommended topic** on Learn — the
engine now offers that newly-unlocked topic. This is the "adaptive path"
story in one tap.

### Beat 6 · "The summary view" (30 s)
Open **Progress**. Point at the three counters (mastered / learning /
untouched), the chapter rollup bars, and the per-topic rows. Close by
saying: "this view is a direct projection of the event log — wipe the
app and replay the log, you get the exact same state."

## 6. Resetting demo state between runs

There is no built-in "reset" button (on purpose — a demo app shouldn't
invite tapping it mid-pitch). Two options:

**Option A — per-student reset (preferred):** in Expo Go, shake the
phone → "Clear AsyncStorage". Next launch, `useBootstrap()` mints a fresh
`student_id` and the event log is empty.

**Option B — uninstall Expo Go and reinstall.** Nuclear but reliable.

For web: open DevTools → Application → Local Storage → clear
`ogrendiem:*` keys → reload.

If you want to rehearse with a *pre-populated* state (some topics already
mastered, to jump straight to Beat 5), add a one-off dev helper:

```ts
// in App.tsx, behind an __DEV__ guard, before useBootstrap():
import { appendEvent } from '@/store/persist';
async function seedDemo(sid: string) {
  const base = Date.now() - 1000 * 60 * 60;
  for (const tid of ['ch1_s1_t1', 'ch1_s1_t2', 'ch1_s2_t1']) {
    for (let i = 0; i < 5; i++) {
      await appendEvent(sid, {
        t: 'answer', ts: base + i * 1000,
        student_id: sid, topic_id: tid,
        question_id: `${tid}__seed__${i}`,
        correct: true, time_ms: 12000,
      });
    }
  }
}
```

Call it once, then delete the call — you now have a "partially-advanced
student" persisted locally.

## 7. Adding more bundled questions

Bundled questions live in `assets/data/questions.json` — a plain array
of `Question` objects (see `src/shared/types.ts`). To add more:

1. Open `src/llm-prompts/worked-example.md` and fill the variables for
   the target topic (topic id, role, tier, n).
2. Paste the filled prompt + `system.md` into your LLM of choice.
3. Validate the LLM's output against `src/llm-prompts/schema.ts`:

   ```ts
   import { questionBatchSchema } from '@/llm-prompts/schema';
   const batch = questionBatchSchema.parse(JSON.parse(llmReply));
   ```

4. Append the parsed items to `assets/data/questions.json`.
5. Restart Metro — Metro bundles JSON at build time, so hot reload
   won't pick up new questions without a restart.

Origin should be `"bundled"` for hand-vetted items, `"llm-generated"`
for raw LLM output (ideally flagged in UI once reviewers are in the loop).

## 8. Key files to know when presenting

| Path | What it does |
|---|---|
| `src/engine/local.ts` | The whole adaptive engine: replay log → derived state, frontier selection, BKT updates. |
| `src/engine/bkt.ts` | 4-parameter BKT (`pLearn 0.25, pSlip 0.1, pGuess 0.2, pForget 0`). |
| `src/engine/propagate.ts` | Noisy-AND: `ability(T) = local(T) · ∏ ability(parents)`. |
| `src/engine/index.ts` | The swap point: `export const engine: TutorEngine = new LocalEngine()`. |
| `src/shared/api.ts` | The `TutorEngine` interface — the contract a future backend must honour. |
| `src/store/persist.ts` | AsyncStorage-backed append-only event log. |
| `assets/data/topics.json` | 55 topics with emoji/slot/cluster baked in. |
| `ogrendiem/graph/export_mobile.py` | The Python exporter that produces those JSON files. |

If a judge asks "where's the magic?" — open `src/engine/local.ts`. It's
~200 lines and it *is* the magic.

## 9. Going from demo to production

The architecture is deliberately one swap away from a real backend:

1. Write a FastAPI server that mirrors `TutorEngine` method signatures
   (there's a Python pydantic type for every TS type in `shared/types.ts`
   — keep them in lockstep).
2. Write `src/engine/remote.ts` that `fetch()`es those endpoints.
3. Change one line in `src/engine/index.ts`:

   ```ts
   export const engine: TutorEngine = new RemoteEngine(API_BASE);
   ```

No screen, component, or store has to change. The event log becomes a
cross-device sync channel (POST events on write, GET derived state on
refresh). LLM-generated questions flow through the backend cache so
two students studying the same topic share the same freshly-authored
items.

## 10. Known rough edges (be honest with the judges)

- **Self-graded answers.** The student self-reports ✓/✗. Auto-grading
  symbolic math is future scope. `question.checks` exists as the hook.
- **BKT `pForget = 0`.** No spaced-repetition decay yet; ignored until
  session length makes it matter.
- **Bundled 30 questions.** 10 topics × 3 roles. The other 45 topics
  route through Learn with "no bundled questions" — they'll be served
  by the on-demand path once the backend exists.
- **Scope is hard-wired** to g1-3-8-9. The scope object is ready for
  arbitrary groups, but the UI only exposes this one.
- **KaTeX over CDN.** Works offline after first load (WebView cache) but
  not on a cold fresh install with no Internet. Swap the CDN URLs in
  `src/components/MathView.tsx` for a bundled local copy if you need
  true airplane-mode guarantees.
